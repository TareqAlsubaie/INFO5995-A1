# INFO5995-A1

# PoC Materials

## Core Vulnerability
  The APK uses the following code to create a session token: `java.util.Random` in:
  - `Login.generateSessionToken()`
  Relevant code references:
  - `Login.java:174`
  - `Login.java:175`
  - `Login.java:183`
  - `Login.java:184`
  - `Login.java:187`

## What The PoC Demonstrates
  - The app creates a session token after successful login
  - The token is generated by a non-cryptographically secure random number generator
  - The token is used as local authentication/session state
  - This makes the design unsuitable for secure session management

## Threat Model Used
  A realistic attacker is a rooted-device user, forensic examiner, or malware with elevated local access who can:
  - reverse engineer the APK
  - inspect decompiled code
  - read or modify local app storage
  - understand the token generation pattern

  ## Limitations
  - These materials do not include a live exploit against a real system
  - The demonstration is analytical and educational
  - Claims were validated by static analysis of the provided APK

  ## Validation
  We manually checked the decompiled code and matched all claims to the APK source:
  - session generation in `Login.generateSessionToken()`
  - session persistence in `SharedPreferences`
  - use of `java.util.Random`

 ## Fix Summary
  A secure mitigation is to replace `java.util.Random` with `java.security.SecureRandom` and the token should be 32 random bytes Base64-encoded.
