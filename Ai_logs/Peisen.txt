High-Level Summary of AI-Assisted Workflow

 

1. Decompilation and workflow setup

- AI was asked which tools are commonly used to inspect an Android APK and how to set up a basic workflow for static analysis.

- AI suggested using tools such as `jadx` and APK decompilation workflows to inspect the manifest, activities, and Java code.

- The group then manually decompiled the APK and inspected authentication-related classes in the decompiled source.

 

2. App understanding and system model support

- AI was asked how to describe an Android app’s system model in a concise security-analysis format.

- AI suggested focusing on components, assets, data flows, attacker assumptions, and trust boundaries.

- The group used that structure to map `MainActivity`, `Login`, `Profile`, local file storage, and `SharedPreferences`.

- The final system model was based on manual reading of the decompiled app logic, not on AI guesses.

 

3. Vulnerability discovery support

- AI was asked how to systematically search for randomness-related issues in an unfamiliar codebase.

- AI suggested searching for values treated as random numbers, tokens, session values, identifiers, codes, or keys, and then checking the API used to generate them.

- Based on this guidance, the group manually searched the decompiled code and located `Login.generateSessionToken()`.

- The group then manually verified that the token was generated using `java.util.Random`.

 

4. Scope checking against the assignment

- AI was asked to help interpret the assignment scope and rubric.

- AI highlighted that Assignment 1 is marked only on randomness and cryptography issues in the provided APK.

- During drafting, broader issues such as plaintext credential storage were discussed, but the group manually checked the specification again and kept the final presentation focus on the session-token randomness issue because it is the clearest in-scope vulnerability.

 

5. Vulnerability explanation and mitigation support

- AI was asked why `java.util.Random` is unsuitable for session tokens and what a sound mitigation would look like.

- AI suggested that session tokens are security-sensitive values and should be generated using a cryptographically strong API such as `SecureRandom`.

- The group manually checked that the selected mitigation matched the vulnerability and was technically appropriate for the app design.

 

6. Presentation and script support

- AI was used to help structure the 5-minute presentation so it clearly covered the system model, threat model, vulnerability, attack scenario, and mitigation.

- AI was also used to help draft a speaking script and slide outline that matched the rubric.

- The group manually revised these outputs to fit the actual code evidence and the speaking-time constraints.

 

7. Activity log and document-format support

- AI was used to help format the activity log, generate a contribution matrix structure, and convert markdown content into `.docx` and `.pdf` formats for submission support.

- The group manually checked names, contribution wording, and consistency across files before finalising them.
