Entry 5: Identifying candidate randomness-related values
 

Input prompt

 

When auditing a decompiled APK for randomness or crypto misuse, what values should I look for and how should I classify them so I can decide which ones are security-sensitive?

 

Output summary

 

The AI suggested searching for tokens, session IDs, verification codes, reset codes, keys, nonces, salts, random strings, and helper methods using APIs such as java.util.Random or SecureRandom. It recommended recording the class and method location, the generator used, and the security role of each value, such as UI-only value, identifier, token, or authentication state.

 

How we used/validated it

 

We used this as a checklist while tracing the app logic. That process led us to generateSessionToken() in the Login class as the key randomness-related method. We confirmed that the generated token was not cosmetic because it was used as stored session state.

 

Entry 6: Assessing the discovered token generation
 

Input prompt

 

In a decompiled Android app, I found a method called generateSessionToken() in the Login class. It builds a 16-character alphanumeric token using java.util.Random and stores it in SharedPreferences as the logged-in session value. Is this a valid security vulnerability for a randomness/crypto assignment? Explain why or why not.

 

Output summary

 

The AI said this is a valid vulnerability because java.util.Random is a general-purpose pseudo-random generator and not appropriate for security-sensitive values such as authentication tokens. It explained that the security issue is not that the token merely "looks random," but that the app assigns a security role to a value generated by a predictable, non-cryptographic API.

 

How we used/validated it

 

We manually checked the code path and confirmed the token’s security role in the authentication flow. This became the core vulnerability in both the report and presentation: insecure session token generation using weak randomness.

 

Entry 7: Turning the weakness into a realistic threat model
 

Input prompt

 

Help me write a realistic attacker model for this case. The attacker can reverse-engineer the APK, understand the token generation logic, and may have local access to app storage on a rooted device or through privileged malware. What concrete attack scenario follows from that?

 

Output summary

 

The AI suggested a rooted-device or privileged-local attacker who can inspect the app, learn the token format and generation logic, and read or modify locally stored session data. It proposed an attack scenario where the attacker decompiles the APK, studies the token creation method, gains local storage access, and manipulates or injects session state to impersonate an authenticated user.

 

How we used/validated it

 

We adopted this attacker model because it matched the locally stored session design of the app and did not rely on unrealistic remote capabilities. We used it in the system/threat model, vulnerability explanation, and attack scenario sections.

 

Entry 8: Asking for a technically sound fix
 

Input prompt

 

What is the most appropriate fix if an Android app uses java.util.Random to generate a session token that represents authenticated state? Please explain both the code-level fix and any higher-level design advice.

 

Output summary

 

The AI recommended replacing java.util.Random with SecureRandom so the token is generated with cryptographically strong unpredictability. It also warned that client-generated and client-trusted session state is a weak design pattern, so a stronger design would avoid trusting a locally generated token as the sole basis for authentication state.

 

How we used/validated it

 

We used this directly in the mitigation section of the report and slide deck. We also cross-checked the fix against standard Java/Android security guidance before including it.
